# Анализ текущего проекта

## 1. Короткий вывод

Текущий репозиторий уже можно считать не просто "dashboard builder", а зачатком внутренней BI-платформы. В нем есть:

- единый fullstack-контур на SvelteKit;
- переиспользуемый BFF-слой для аналитических датасетов;
- декларативная система фильтров;
- общий UI-kit и шаблоны визуализации;
- прикладные модули на PostgreSQL;
- фоновый scheduler для алертов.

Это хорошая база для EMIS, особенно если нам нужен единый продукт, где BI и картографическая аналитика живут рядом. Но как платформа для параллельного роста двух направлений проекту все еще не хватает дисциплины на уровне документации, модульных границ, миграций, тестов и управляемого старта инфраструктурных частей.

Главная рекомендация: не делать резкий физический split в монорепо прямо сейчас. Сначала стоит стабилизировать текущий репозиторий как "platform-ready single app", а EMIS запускать как отдельный домен внутри него с границами, совместимыми с будущим монорепо.

### Update от 16 марта 2026

Часть рисков, описанных ниже, уже частично закрыта:

- filter lifecycle для active Wildberries routes и `/emis` больше не держится только на top-level `registerFilters(...)`;
- в проекте появился workspace-aware runtime (`useFilterWorkspace(...)`) с URL sync и target-aware planning;
- `/emis` уже стал рабочим map/list workspace;
- блок про accessibility warnings в filter widgets ниже устарел: после последних правок `pnpm check` уже проходит без этих warning.

### Update от 17 марта 2026

Для части проекта вне EMIS дополнительно зафиксирована platform policy:

- source-agnostic путь `fetchDataset -> /api/datasets/:id -> compileDataset -> provider` сохраняется как canonical contract для BI/read-side;
- это считается осознанной платформенной абстракцией, а не переусложнением, пока scope ограничен non-EMIS analytical flows;
- alerts, proxy endpoints и EMIS operational/query transport в это правило не включаются.

## 2. Что представляет собой проект сейчас

### Текущий профиль проекта

- Один пакет SvelteKit, без реального разбиения на `apps/*` и `packages/*`
- UI, BFF и server-side логика живут в одном приложении
- Есть both mock и PostgreSQL-backed сценарии
- Есть прикладной контур Wildberries, который показывает, что проект умеет работать не только как UI demo, но и как аналитическое приложение

### Ключевые слои

1. `src/lib/shared/*`
   Общие UI-примитивы, chart wrappers, стили, утилиты, API facade.

2. `src/lib/entities/dataset/*`
   Контракты датасетов и промежуточное IR для абстракции над источниками данных.

3. `src/lib/entities/filter/*`
   Декларативная система фильтров, registry и store.

4. `src/lib/server/*`
   BFF-слой: compilation, providers, DB access, alerts.

5. `src/lib/features/dashboard-edit/*`
   Редактор дашбордов на GridStack.

6. `src/routes/dashboard/wildberries/*`
   Реальные прикладные страницы и расчетная аналитика.

## 3. Что в проекте уже сильное

### 3.1. Правильный fullstack-контур для аналитики

Паттерн `fetchDataset -> /api/datasets/:id -> compileDataset -> provider` уже задает хороший каркас для аналитических read-моделей. Это особенно полезно для EMIS, где будут:

- BI-срезы поверх оперативных данных;
- documentированные views/read models;
- табличные аналитические контракты, которые действительно должны жить в общей BI abstraction.

Сильная сторона в том, что UI не знает ничего о SQL и источнике данных, а сервер уже умеет держать устойчивый контракт. Но это стоит сохранять именно как BI/read-side contract, а не распространять автоматически на EMIS map/search/detail flows.

### 3.2. Неплохая модульность внутри одного приложения

Несмотря на отсутствие физического монорепо, проект уже мыслится как модульная система:

- shared;
- entities;
- features;
- widgets;
- server.

Для старта EMIS это лучше, чем начинать с чистого листа.

### 3.3. Переиспользуемые фильтры и client/server planning

Система фильтров - одна из самых полезных заготовок для EMIS. Она уже умеет:

- хранить единое состояние фильтров;
- регистрировать фильтры декларативно;
- планировать, что уйдет на сервер, а что останется на клиенте;
- связывать один фильтр с несколькими датасетами.

Для сценария "карта + список + карточка + BI-виджет" это очень ценно.

### 3.4. Есть реальные доменные сценарии, а не только UI-kit

Wildberries-модули показывают, что команда уже идет дальше витрины компонентов:

- есть работа с PostgreSQL;
- есть отдельные page-level calculations;
- есть API-прокси к внешнему сервису;
- есть алертинг.

Это снижает риск при запуске EMIS: базовые организационные паттерны уже нащупаны.

## 4. Что ограничивает проект как основу для EMIS

### 4.1. Документация отстает от фактического состояния

Самая заметная проблема сейчас - верхнеуровневая документация не соответствует текущему состоянию проекта.

- `README.md` до правок был фактически стандартным шаблоном Svelte.
- На уровне входной точки проекта все еще остались scaffold-элементы (`Header`, footer, demo-root page), которые не объясняют, что это уже BI-платформа.

Следствие: новому участнику трудно понять, где demo, где платформа, а где прикладная логика.

### 4.2. Проект еще не готов как "общая платформа на двоих"

Сейчас код живет в одном приложении, но без формализованного разделения на:

- platform/shared;
- текущий BI-домен;
- будущий EMIS-домен.

Это не критично для текущего масштаба, но при параллельной разработке BI и EMIS быстро начнет мешать:

- сложнее ревьюить изменения;
- тяжелее согласовывать shared contracts;
- выше риск случайной доменной связанности.

### 4.3. Data/BFF слой хороший, но пока еще MVP по зрелости

Ограничения, которые уже видно по коду:

- provider routing хардкожен по префиксу dataset id;
- registry датасетов собирается вручную;
- SQL provider покрывает только ограниченный поднабор IR;
- тестового контура вокруг compile/provider/filter planning пока нет.

Для EMIS это значит, что read-модели можно переиспользовать, но сначала желательно усилить платформенный слой.

### 4.4. Filter lifecycle был проблемой и частично закрыт, но legacy-хвост еще остается

Изначально на страницах фильтры регистрировались через `registerFilters(...)`, и это создавало явные риски:

- `unregisterFilters(...)` не используется;
- page-scoped filters пока фактически не заведены в рабочий цикл;
- `setActivePage(...)` в реальных страницах не применяется.

После platform upgrade часть этого уже исправлена:

- active Wildberries routes и `/emis` используют `useFilterWorkspace(...)`;
- lifecycle теперь закрывается через mount/unmount cleanup;
- URL sync и target-aware planner уже работают в реальном коде.

Но риск еще не исчез полностью:

- legacy API и старый singleton-режим все еще остаются в кодовой базе;
- не все маршруты проекта переведены на новый runtime;
- cleanup старых подходов пока намеренно отложен.

### 4.5. Операционный контур пока фрагментарный

В проекте уже есть scheduler alerts, но пока нет полноценного platform baseline:

- snapshot-first source of truth по основной бизнес-схеме;
- seed-процесса как стандарта;
- единых smoke/integration тестов;
- явного разделения boot-процессов app/runtime/background tasks.

Отдельно стоит отметить, что scheduler стартует на уровне `hooks.server.ts`, и это заметно даже во время production build. Для будущего EMIS это лучше сделать управляемым и явным.

### 4.6. Качество baseline пока неровное

По быстрой проверке состояния проекта на 14 марта 2026:

- `pnpm check` проходит без ошибок, но есть 4 warning по accessibility в filter widgets;
- `pnpm build` проходит, но есть warning по `useDebouncedLoader`, accessibility warnings и предупреждение о крупном client chunk;
- `pnpm lint` падает, потому что в проекте накопился большой Prettier drift.

Это не блокер для анализа, но показатель: перед масштабированием платформы хорошо бы стабилизировать baseline.

## 6. Что стоит исправить в текущем проекте до или параллельно старту EMIS

### Приоритет 1. Сформировать платформенную рамку

Нужно зафиксировать, что в репозитории есть:

- текущая BI-платформа;
- shared/platform слой;
- будущий EMIS как новый домен.

Практически это означает:

- привести README и docs в соответствие проекту;
- определить, какие папки считаются platform/shared;
- завести отдельное пространство имен под EMIS (`src/lib/domains/emis`, `src/routes/emis`, `src/lib/server/emis` или эквивалент).

### Приоритет 2. Усилить baseline данных

Для EMIS потребуется не только чтение, но и запись. Поэтому до активной разработки EMIS нужно подготовить:

- snapshot baseline и documented patch workflow для основной БД;
- seed-механизм;
- repository/service слой для CRUD;
- минимальные integration tests на ключевые data flows.

### Приоритет 3. Нормализовать lifecycle фильтров

До старта map-heavy домена стоит закрыть:

- unregister filters на unmount;
- page scope;
- явную активную страницу;
- правила совместного использования filters между list/map/inspector.

Update от 16 марта 2026:

- этот приоритет уже частично закрыт для active routes;
- следующая задача здесь не "изобрести runtime", а дочистить legacy paths и унифицировать remaining pages.

### Приоритет 4. Развести app boot и background jobs

Scheduler полезен, но для платформы лучше, чтобы:

- build/test/dev работали предсказуемо;
- background jobs поднимались управляемо;
- side effects не происходили просто от импорта runtime-модуля.

### Приоритет 5. Подчистить UX-обвязку корневого приложения

Сейчас верхний shell проекта еще носит demo-характер. До расширения под EMIS стоит:

- привести root navigation к осмысленному виду;
- отделить demo playground от реальных прикладных модулей;
- определить будущий единый app shell для BI и EMIS.

## 7. Рекомендация по архитектурной траектории

### Рекомендуемый вариант на ближайший этап

Не делать резкий переход в физическое монорепо прямо сейчас.

Лучше идти в два шага:

1. Сохранить один deployable SvelteKit app.
2. Внутри него построить монорепо-совместимые доменные границы.

Это даст:

- быстрый старт EMIS без инфраструктурного overhead;
- переиспользование уже готовых platform-модулей;
- понятную точку для будущего split на `apps/*` и `packages/*`.

### Когда монорепо действительно станет нужно

Физический split в монорепо стоит делать, когда появится хотя бы один из признаков:

- у BI и EMIS разный release cadence;
- появляется отдельная команда/подкоманда на EMIS;
- у EMIS своя тяжелая карта/geo-инфраструктура и отдельный deployment lifecycle;
- shared code перестает помещаться в один app без организационного шума;
- нужен отдельный runtime для ingestion/background pipelines.

## 8. Предлагаемое разбиение работы

### Этап A. Стабилизация текущей платформы

- обновить верхнеуровневую документацию;
- зафиксировать доменные и platform boundaries;
- определить будущий app shell;
- описать правила shared/reuse;
- принять решение, в каком namespace живет EMIS внутри текущего приложения.

### Этап B. Архитектурный фундамент EMIS

- согласовать обновленное ТЗ;
- сформировать data model;
- выбрать стратегию snapshot/seed и optional delta patch;
- определить write API и read-model API;
- выделить geo/map слой.

### Этап C. Первая рабочая вертикаль EMIS

- справочники;
- реестр объектов;
- карта объектов;
- базовые фильтры;
- карточка объекта.

### Этап D. Новости и связи

- новости;
- источники;
- связи news <-> objects;
- общая карта и синхронные фильтры.

### Этап E. BI readiness

- SQL views/read models;
- аналитические витрины;
- встроенные BI-срезы по данным EMIS;
- точка решения: единое приложение или физическое монорепо.
