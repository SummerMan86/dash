# Brainstorming Protocol

Интерактивный problem-framing playbook для `lead-strategic`. Применяется до написания `current_plan.md` на non-trivial product-задачах. Цель — убедиться, что мы решаем правильную задачу, а не красиво реализуем неправильную.

## When to use

Запускай brainstorming, если выполняется **хотя бы одно** из условий:

- постановка задачи широкая или неоднозначная;
- unfamiliar domain для текущего контекста (новая страница, новый provider, новый dataset type);
- задача вводит новый источник данных, новую BI-страницу или новый widget type;
- есть несколько валидных архитектурных подходов (client-only vs BFF, новый provider vs переиспользование existing);
- acceptance criteria не очевидны из постановки;
- ожидается cross-layer или cross-package touch;
- задача попадает в зону известного migration debt (см. `bi/architecture.md` §9);
- пользователь явно попросил «давай сначала обсудим».

## When to skip

Пропускай brainstorming, если:

- задача подпадает под `direct-fix` (`<= 10` строк, один файл, нет architectural surface);
- задача явно batch / low-risk с понятным scope;
- задача — продолжение уже сформулированной волны (brainstorm был на входе волны);
- изменение чисто механическое (rename, formatting, trivial bug fix, docs-only typo);
- пользователь явно сказал «план без брейнсторма».

Skip всегда фиксируется в `current_plan.md` одной строкой: `Brainstorm: skipped — <причина>`. Silent skip не допускается.

## Protocol

Brainstorming ведётся интерактивно с пользователем. В autonomous mode — **skill не применяется**: в autonomous пользователь недоступен как собеседник, а brainstorm без него превращается в угадывание. В autonomous mode `lead-strategic` идёт сразу к плану и фиксирует `Brainstorm: skipped — autonomous mode`.

### 1. Переформулируй задачу своими словами

- Запиши, как ты понял задачу (1-3 предложения).
- Явно выдели implicit assumptions (что ты домыслил, чего в постановке не было).
- Покажи пользователю и попроси подтвердить или поправить.

Это первый checkpoint. Если понимание разошлось с пользователем — дальше не идём, пока не сходится.

### 2. Уточняющие вопросы маленькими чанками

- Не больше **3-5 вопросов за один turn**. Двадцать вопросов простынёй — anti-pattern: пользователь устаёт, отвечает поверхностно.
- Порядок приоритета: **scope → acceptance → constraints → integrations → alternatives**.
- После каждого chunk'а жди ответов, только потом следующий.
- Если пользователь сам закрыл тему ответом «понятно, дальше план» — остановись и переходи к §6.

Типовые категории вопросов:

- **Scope:** что точно входит, что точно вне. MVP vs nice-to-have. Какие user flows покрываем в этой волне.
- **Acceptance:** как мы узнаем что готово? Observable behavior, не внутренние детали.
- **Constraints:** кто видит эту фичу (роли, tenant scope), ограничения на performance, данные.
- **Integrations:** переиспользуем существующие datasets / providers / фильтры, или нужны новые?
- **Alternatives:** есть ли 2-3 плоских варианта подхода, которые стоит обсудить до коммита.

### 3. Проверь пересечение с существующим

Перед тем как предлагать новое — посмотри, не закрыто ли часть задачи уже:

- existing datasets в registry;
- existing providers (postgres, mock);
- existing widgets и их capabilities;
- existing filter scopes (`shared` / `workspace` / `owner`);
- migration debt в зоне задачи (см. `bi/architecture.md` §9 и `invariants.md`).

Если пересечение есть — озвучь его пользователю и спроси, переиспользуем или делаем новое.

### 4. Surface alternatives

Если задача допускает несколько подходов — назови **2-3** плоских варианта, по каждому одну строку tradeoff. Не нужно глубокого анализа каждого — это brainstorm, не ADR. Цель — чтобы пользователь понял веер и выбрал направление до плана.

Пример для «добавить страницу с новым источником данных»:

1. **Новый provider + новый dataset type** — максимальная изоляция, но расширяет provider registry.
2. **Переиспользовать postgresProvider + новый dataset** — дешевле, если источник SQL-совместим.
3. **Client-only transformation на existing dataset** — годится, если данные уже есть, просто нужна новая проекция.

### 5. Явно спроси про non-goals

Что мы **не делаем** в этой волне? Это важнее, чем кажется: non-goals защищают от scope drift. Зафиксируй 2-5 явных non-goals.

### 6. Зафиксируй summary и получи подтверждение

После того как вопросы закрыты, отдай пользователю **краткий summary** (максимум 10-15 строк):

- что делаем;
- что не делаем;
- какой подход выбран (если были альтернативы);
- какие остались открытые вопросы, которые требуют решения в plan;
- явный риск-флаг (если есть).

Пользователь говорит `go` / `ок, план`, или возвращает правки. Только после `go` переходим к написанию `current_plan.md`.

## Output

Brainstorm **не создаёт отдельный артефакт**. Результат живёт внутри `current_plan.md` как набор секций (в начале файла, до декомпозиции на slices):

- **Task Understanding** — переформулированная задача, подтверждённое пользователем понимание.
- **Scope / Non-goals** — что в волне и что точно вне.
- **Chosen Approach** — выбранный подход из альтернатив, с one-line rationale. Если альтернатив не было — одна строка «single approach, no alternatives considered».
- **Open Questions** — уточнения, которые не получилось закрыть в brainstorm; должны быть resolved до старта первого slice (через architecture pass, transparency request или дополнительный вопрос пользователю).
- **Risk Flags** — инварианты в зоне задачи, migration debt, unfamiliar code.

Далее идёт обычная декомпозиция slices по `templates.md §3`.

## Interactive discipline

- Маленькие чанки (3-5 вопросов за turn), не простыни.
- Никакого plan writing до явного `go` от пользователя.
- Пользователь в любой момент может прервать brainstorm фразой «хватит, пиши план» — это валидный выход.
- Если пользователь отвечает односложно и явно не хочет углубляться — не дави, переходи к summary с тем что есть, open questions зафиксируй явно.
- Не повторяй уже полученные ответы под видом «уточнения».

## Escalation triggers

Остановись и эскалируй, если в процессе brainstorm'а обнаружилось:

- задача требует product / business decision, не технический выбор → продолжать brainstorm не твоя зона, финализируй summary и передай пользователю;
- задача конфликтует с существующим инвариантом (`invariants.md` или overlay) → `architecture pass` как prerequisite slice, не plan сразу;
- scope слишком широкий даже после уточнения → предложи разбить на несколько волн, brainstorm каждую отдельно;
- пользователь не может дать acceptance criteria → это сигнал, что задача не созрела, предложи отложить или сделать exploration spike.

## Anti-patterns

- **Questionnaire dump** — 20 вопросов списком в первом же сообщении.
- **Brainstorm как замена плана** — глубокий анализ каждой альтернативы вместо декомпозиции.
- **Silent skip** — проигнорировал skill на non-trivial задаче без записи причины в `current_plan.md`.
- **Brainstorm без summary** — пообщались и сразу в plan, пользователь не успел подтвердить понимание.
- **Решения без пользователя** — сам закрыл open question, не спросив, потому что «кажется очевидным».
- **Brainstorm в autonomous mode** — в autonomous нет пользователя-собеседника, skill там неприменим.
