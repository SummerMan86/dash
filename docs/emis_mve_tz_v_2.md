# ТЗ на MVE EMIS v2

## 1. Статус документа

Этот документ является обновленной редакцией `emis_mve_tz_v_1.md` и учитывает:

- текущее состояние репозитория `dashboard-builder`;
- возможность развивать EMIS как часть единого приложения вместе с BI;
- альтернативный вариант последующего перехода в монорепо;
- необходимость не просто "сделать карту", а заложить устойчивое производственное ядро.

## 2. Контекст

Сейчас у команды уже есть SvelteKit-приложение, которое фактически выступает как BI/fullstack-платформа:

- общий UI-kit;
- BFF-слой для аналитических датасетов;
- фильтры;
- PostgreSQL-backed аналитические страницы;
- server-side фоновые задачи.

EMIS должен проектироваться не как изолированный pet-project, а как новый домен поверх этой базы.

При этом на уровне product topology допускаются два сценария:

1. BI и EMIS остаются частями одного приложения.
2. BI и EMIS становятся разными приложениями внутри общего монорепо.

## 3. Главная идея MVE

Для первой очереди EMIS принимается подход **MVE (Minimum Viable Engine)**.

Под MVE в данном документе понимается:

- минимальный, но рабочий контур для объектов, новостей и картографической аналитики;
- архитектура, пригодная для роста;
- явное разделение operational CRUD и analytical read-моделей;
- совместимость с текущим BI-контуром;
- отсутствие преждевременного усложнения инфраструктуры.

## 4. Ключевые архитектурные решения

Заранее фиксируются следующие решения.

1. Первая версия EMIS реализуется как **модульный монолит**, а не как набор микросервисов.
2. На уровне продукта MVE должен быть **одним deployable fullstack application**.
3. На уровне кода MVE должен быть **monorepo-ready**, даже если физически сначала остается в одном приложении.
4. **PostgreSQL + PostGIS** используются с первого этапа.
5. Для записи и управления сущностями используется обычный application/backend слой.
6. Для аналитических списков, картографических выборок и BI-срезов допускается отдельный read/query слой.
7. Схема данных изначально проектируется с прицелом на дальнейшее чтение из BI и аналитических витрин.

## 5. Что именно мы рекомендуем по topology

### 5.1. Базовая рекомендация

На ближайший этап рекомендуется:

- не делать сразу отдельный репозиторий;
- не начинать с полного физического split в `apps/*` и `packages/*`;
- строить EMIS как новый домен рядом с текущим BI-контуром внутри существующего проекта;
- соблюдать такие границы, чтобы later split в монорепо был недорогим.

Иными словами:

- **deployment now**: единое приложение;
- **code organization now**: модульные домены и platform/shared слой;
- **future option**: монорепо без переписывания бизнес-логики.

### 5.2. Допустимый альтернативный вариант

Если команда заранее уверена, что BI и EMIS будут быстро расходиться по release cadence и ownership, можно сразу строить физическое монорепо.

Этот вариант допустим, но не является обязательным требованием MVE.

### 5.3. Когда переход в физическое монорепо оправдан

Переход оправдан, когда появляется хотя бы один из критериев:

- отдельная команда на EMIS;
- отдельный deployment lifecycle;
- отдельные background pipelines/ingestion runtime;
- существенный рост shared packages;
- заметное расхождение product roadmap между BI и EMIS.

## 6. Цель первой версии

Первая версия EMIS должна обеспечить:

1. Единый реестр инфраструктурных объектов.
2. Единое хранилище новостей и источников.
3. Устойчивую связь новостей с объектами.
4. Единый интерфейс "карта + список + карточка".
5. Базовые фильтры и поиск.
6. Подготовленные read-модели и SQL views для аналитики и BI.
7. Возможность дальнейшего расширения в сторону:
   - событий;
   - ingestion pipelines;
   - верификации;
   - AI/NLP;
   - специализированных аналитических дашбордов.

## 7. Границы MVE

### 7.1. Что входит в первую версию

#### Operational контур

- справочники стран, типов объектов и источников;
- ручное создание и редактирование объектов;
- ручное создание и редактирование новостей;
- ручное создание и редактирование связей между новостями и объектами;
- мягкое удаление объектов и новостей;
- seed/test данные и dev-reset сценарии.

#### Query / analytical контур

- список объектов;
- список новостей;
- карточка объекта;
- карточка новости;
- карта объектов;
- карта новостей при наличии координат;
- совмещенный экран с картой и результатами поиска;
- фильтры и поиск;
- bbox/viewport-фильтрация;
- SQL views/read models для BI.

#### Platform контур

- миграции БД;
- локальный запуск через Docker Compose;
- стабильные API-контракты;
- базовая диагностика состояния приложения;
- минимальные тесты на критические data flows.

### 7.2. Что не входит в первую версию

На первом этапе не входят:

- автоматический ingestion новостей из внешних источников;
- Kafka, Redis, Elasticsearch, MinIO;
- LLM/NLP, дедупликация, авто-классификация;
- отдельный Python backend;
- сложная RBAC-модель;
- real-time streaming;
- импорт Excel/GeoJSON через UI;
- AIS/трековые данные;
- отдельная обязательная сущность `events`;
- полноценный пользовательский конструктор аналитических дашбордов.

### 7.3. Что должно быть готово архитектурно

Даже если это не входит в MVE, архитектура должна допускать:

- добавление `events`;
- добавление ingestion pipeline;
- добавление verification workflow;
- вынос background jobs в отдельный runtime;
- вынос EMIS в отдельное приложение внутри монорепо;
- безопасное чтение данных из BI-инструментов.

## 8. Связь с текущим проектом

### 8.1. Что EMIS должен переиспользовать

- shared UI и design tokens;
- паттерн server-side BFF;
- общие типы DTO и контракты;
- общую систему фильтров после ее стабилизации;
- PostgreSQL integration patterns;
- инфраструктурные подходы к alerts/notifications при необходимости.

### 8.2. Что не нужно механически переносить

- текущие demo-страницы;
- Wildberries-доменную логику;
- локальные MVP-упрощения редактора дашбордов;
- произвольные route names и demo-shell текущего проекта.

## 9. Архитектурный принцип EMIS

EMIS строится как **модульный монолит с разделением на platform и domain modules**.

Рекомендуемые доменные модули:

- `objects`
- `news`
- `sources`
- `links`
- `geo`
- `dictionaries`
- `read-models`
- `notifications` или `alerts` при необходимости

Ключевой принцип:

- write side не зависит от UI;
- read side не должен повторять UI-структуру страниц;
- shared/platform слой не должен зависеть от EMIS-домена.

## 10. Рекомендуемая организация кода

### 10.1. Если EMIS остается в текущем приложении

Рекомендуемая схема:

```text
src/
  lib/
    platform/
      shared-ui/
      shared-styles/
      dataset/
      filters/
      db/
      auth/
    domains/
      bi/
      emis/
        objects/
        news/
        sources/
        links/
        geo/
        read-models/
  routes/
    dashboard/
    emis/
    api/
      datasets/
      emis/
```

Смысл этой схемы:

- shared и platform выносятся в явный слой;
- BI и EMIS получают отдельные доменные пространства;
- приложение по-прежнему остается единым.

### 10.2. Если принимается физическое монорепо

Рекомендуемая схема:

```text
repo/
  apps/
    bi/
    emis-web/
  packages/
    db/
    shared-types/
    shared-ui/
    shared-config/
    platform-dataset/
    platform-filters/
  docs/
  scripts/
```

### 10.3. Что считается обязательным в обоих вариантах

Независимо от topology, должны быть отделены:

- platform/shared UI;
- DB schema и migrations;
- DTO/contracts;
- read-model/query layer;
- domain services для CRUD;
- EMIS routes и EMIS API namespace.

## 11. Технологический стек

### Обязательный стек

- Frontend / Fullstack: Svelte 5, SvelteKit 2
- База данных: PostgreSQL 16+
- Геоданные: PostGIS
- Карта: MapLibre GL JS
- Стили: Tailwind CSS
- Тесты: Vitest
- Контейнеризация: Docker Compose

### SQL / data access слой

Допускается один из вариантов:

- Kysely;
- Drizzle ORM;
- комбинированный подход, где CRUD идет через typed query builder, а аналитические/PostGIS-запросы при необходимости пишутся вручную.

Окончательный выбор фиксируется в Implementation Spec.

## 12. Логическая архитектура решения

### 12.1. Write side

Write side отвечает за:

- создание и редактирование объектов;
- создание и редактирование новостей;
- управление связями news <-> object;
- справочники;
- soft delete;
- валидацию;
- транзакционность.

Этот слой может быть реализован через repositories + services + REST endpoints или server actions внутри SvelteKit.

### 12.2. Read side

Read side отвечает за:

- списки;
- карточки;
- map layers;
- search;
- analytical views;
- BI-compatible read models.

Для read side допускается использовать:

- обычные REST endpoints;
- dataset/read-model endpoints в духе текущего BI/BFF;
- заранее определенные SQL views.

### 12.3. Почему это важно

Для EMIS CRUD и аналитика имеют разную природу:

- CRUD работает с доменными правилами и записью;
- карта и BI работают с быстрыми read-моделями;
- попытка смешать все в одном слое быстро усложнит систему.

Поэтому в MVE принимается **CQRS-lite без микросервисов**:

- один модульный монолит;
- разные ответственности для write и query слоев.

## 13. Доменная модель

В первой версии основными сущностями являются:

- `countries`
- `object_types`
- `objects`
- `sources`
- `news_items`
- `news_object_links`

Дополнительно допускаются:

- `tags` и `news_tags`
- `object_aliases`
- `object_external_refs`

Но только если их реализация не размывает первую очередь.

## 14. Сущность `events`

В первой версии `events` не является обязательной сущностью.

Причины:

- пользователю уже нужна практическая работа с объектами и новостями;
- события потребуют дополнительной логики консолидации и верификации;
- для MVE важнее сначала создать устойчивую operational и geospatial основу.

При этом модель должна позволять позднее добавить:

- `events`
- `event_news_links`
- `event_object_links`

без перелома существующей схемы.

## 15. Модель данных первой версии

### 15.1. `countries`

Справочник стран.

Поля:

- `code CHAR(2) PK`
- `name_ru TEXT NOT NULL`
- `name_en TEXT NOT NULL`

### 15.2. `object_types`

Справочник типов объектов.

Поля:

- `id UUID PK`
- `code TEXT UNIQUE NOT NULL`
- `name TEXT NOT NULL`
- `geometry_kind TEXT NOT NULL`
- `icon_key TEXT NULL`
- `created_at TIMESTAMPTZ NOT NULL`

### 15.3. `objects`

Основной реестр инфраструктурных объектов.

Поля:

- `id UUID PK`
- `external_id TEXT NULL`
- `object_type_id UUID NOT NULL`
- `name TEXT NOT NULL`
- `name_en TEXT NULL`
- `country_code CHAR(2) NULL`
- `region TEXT NULL`
- `status TEXT NOT NULL DEFAULT 'active'`
- `operator_name TEXT NULL`
- `description TEXT NULL`
- `attributes JSONB NOT NULL DEFAULT '{}'::jsonb`
- `geom geometry(Geometry, 4326) NOT NULL`
- `centroid geometry(Point, 4326) NULL`
- `source_note TEXT NULL`
- `created_at TIMESTAMPTZ NOT NULL`
- `updated_at TIMESTAMPTZ NOT NULL`
- `deleted_at TIMESTAMPTZ NULL`

Правило:

`attributes` используется только для вторичных, типо-зависимых полей. Основные атрибуты объекта остаются нормализованными.

### 15.4. `sources`

Справочник источников новостей.

Поля:

- `id UUID PK`
- `code TEXT UNIQUE NOT NULL`
- `name TEXT NOT NULL`
- `kind TEXT NOT NULL`
- `base_url TEXT NULL`
- `is_active BOOLEAN NOT NULL DEFAULT true`
- `created_at TIMESTAMPTZ NOT NULL`
- `updated_at TIMESTAMPTZ NOT NULL`

### 15.5. `news_items`

Основная таблица новостей.

Поля:

- `id UUID PK`
- `source_id UUID NOT NULL`
- `source_item_id TEXT NULL`
- `url TEXT NULL`
- `title TEXT NOT NULL`
- `summary TEXT NULL`
- `body TEXT NULL`
- `language CHAR(2) NULL`
- `published_at TIMESTAMPTZ NOT NULL`
- `collected_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- `country_code CHAR(2) NULL`
- `region TEXT NULL`
- `news_type TEXT NULL`
- `importance SMALLINT NULL`
- `geom geometry(Point, 4326) NULL`
- `is_manual BOOLEAN NOT NULL DEFAULT false`
- `meta JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at TIMESTAMPTZ NOT NULL`
- `updated_at TIMESTAMPTZ NOT NULL`
- `deleted_at TIMESTAMPTZ NULL`

### 15.6. `news_object_links`

Связь many-to-many между новостями и объектами.

Поля:

- `id UUID PK`
- `news_id UUID NOT NULL`
- `object_id UUID NOT NULL`
- `link_type TEXT NOT NULL DEFAULT 'mentioned'`
- `is_primary BOOLEAN NOT NULL DEFAULT false`
- `confidence NUMERIC(5,4) NULL`
- `comment TEXT NULL`
- `created_at TIMESTAMPTZ NOT NULL`
- `UNIQUE(news_id, object_id, link_type)`

### 15.7. Индексы

Обязательные индексы:

- B-tree по `news_items(published_at DESC)`
- B-tree по `news_items(source_id, published_at DESC)`
- B-tree по `objects(object_type_id)`
- B-tree по `objects(country_code)`
- GIST по `objects.geom`
- GIST по `news_items.geom`
- B-tree по `news_object_links(object_id)`
- B-tree по `news_object_links(news_id)`
- полнотекстовый индекс по `news_items(title, summary, body)` через PostgreSQL FTS

### 15.8. BI / analytical views

Нужно предусмотреть как минимум:

- `vw_news_flat`
- `vw_object_news_facts`
- `vw_objects_dim`

#### `vw_news_flat`

Плоское представление новостей:

- id новости
- источник
- дата публикации
- страна
- тип новости
- важность
- количество связанных объектов

#### `vw_object_news_facts`

Факт связи новости и объекта:

- id новости
- id объекта
- тип объекта
- страна объекта
- дата публикации новости
- источник
- тип связи
- primary/non-primary

#### `vw_objects_dim`

Упрощенное измерение объектов для BI:

- id объекта
- тип объекта
- страна
- регион
- статус
- оператор

## 16. Функциональные требования

### 16.1. Модуль объектов

Система должна обеспечивать:

- создание объекта вручную;
- редактирование объекта;
- мягкое удаление объекта;
- просмотр списка объектов;
- поиск по названию;
- фильтрацию по типу, стране, статусу;
- просмотр объекта на карте;
- просмотр карточки объекта;
- отображение связанных новостей в карточке объекта.

### 16.2. Модуль новостей

Система должна обеспечивать:

- загрузку seed/test новостей;
- ручное создание и редактирование новости;
- просмотр списка новостей;
- сортировку по дате публикации;
- фильтрацию по источнику, стране, типу, периоду;
- поиск по заголовку и тексту;
- просмотр карточки новости;
- отображение связанных объектов в карточке новости.

### 16.3. Связи новостей и объектов

Система должна обеспечивать:

- ручное связывание новости с одним или несколькими объектами;
- выбор primary object для новости;
- отображение всех связанных новостей у объекта;
- отображение всех связанных объектов у новости.

### 16.4. Карта

Система должна обеспечивать:

- отображение объектов на карте;
- отображение новостей на карте при наличии координат;
- отдельные слои для объектов и новостей;
- фильтрацию карты по текущим фильтрам;
- bbox / viewport filtering;
- открытие карточки по клику;
- синхронизацию карты и списка результатов.

### 16.5. Аналитический слой первой версии

Система должна обеспечивать:

- подготовленные SQL views/read models;
- минимум один аналитический экран или встраиваемый аналитический блок поверх данных EMIS;
- возможность подключать read-модели EMIS к текущему BI-контурy без перепроектирования БД.

### 16.6. Административные и эксплуатационные функции

Система должна обеспечивать:

- запуск миграций;
- запуск seed данных;
- dev reset базы;
- health/diagnostics endpoint или страница;
- логирование ошибок API.

## 17. UI/UX требования

### 17.1. Базовая модель интерфейса

Для MVE рекомендуется единый рабочий контур:

- карта как центральная часть;
- панель фильтров;
- список результатов;
- переходы в карточки объектов и новостей.

Дополнительные каталожные страницы допустимы, но ключевой ценностью MVE должен быть именно совместный сценарий "карта + аналитический список".

### 17.2. Основные экраны

1. Главный workspace EMIS
2. Каталог объектов
3. Каталог новостей
4. Карточка объекта
5. Карточка новости
6. Административные формы справочников и связей

### 17.3. Если BI и EMIS живут в одном приложении

В этом случае UI должен допускать единый shell:

- общий sidebar/top navigation;
- отдельный раздел `BI`;
- отдельный раздел `EMIS`;
- общие design tokens и shared components;
- отдельные доменные маршруты.

## 18. API первой версии

Все API первой версии реализуются внутри SvelteKit.

Рекомендуется использовать отдельный namespace:

- `/api/emis/*`

Это упростит coexistence с текущими BI endpoints.

### 18.1. Write API

#### Объекты

- `GET /api/emis/objects`
- `GET /api/emis/objects/:id`
- `POST /api/emis/objects`
- `PATCH /api/emis/objects/:id`
- `DELETE /api/emis/objects/:id`

#### Новости

- `GET /api/emis/news`
- `GET /api/emis/news/:id`
- `POST /api/emis/news`
- `PATCH /api/emis/news/:id`
- `DELETE /api/emis/news/:id`

#### Связи

- `POST /api/emis/news/:id/objects`
- `PATCH /api/emis/news/:id/objects/:objectId`
- `DELETE /api/emis/news/:id/objects/:objectId`

#### Справочники

- `GET /api/emis/dictionaries/object-types`
- `GET /api/emis/dictionaries/sources`
- `GET /api/emis/dictionaries/countries`

### 18.2. Read / map API

Рекомендуемые endpoints:

- `GET /api/emis/search/objects`
- `GET /api/emis/search/news`
- `GET /api/emis/map/objects`
- `GET /api/emis/map/news`

Допускается, что часть аналитических read-моделей будет выдаваться через dataset-style endpoints, если это согласуется с текущим платформенным слоем.

### 18.3. GeoJSON

Для карты желательно выдавать GeoJSON:

- `GET /api/emis/map/objects`
- `GET /api/emis/map/news`

## 19. Интеграция с текущим BI

### 19.1. Обязательные условия

1. Схема БД должна быть понятна BI-слою без сложных обходов.
2. Идентификаторы сущностей должны быть стабильными.
3. Должны существовать плоские views/read models.
4. Наименования полей не должны зависеть от UI.
5. EMIS не должен ломать shared platform contracts.

### 19.2. Как именно связывать BI и EMIS

Приоритетный вариант:

- operational данные EMIS живут в PostgreSQL/PostGIS;
- поверх них создаются views/read models;
- BI использует эти views/read models напрямую или через существующий BFF/dataset слой.

Важно:

- BI не должен зависеть от CRUD-форм EMIS;
- EMIS не должен зависеть от конкретных BI-страниц;
- точка интеграции должна быть на уровне schema/views/contracts.

## 20. Нефункциональные требования

### 20.1. Производительность

При объеме до:

- 100 000 новостей;
- 50 000 объектов;
- 500 000 связей;

система должна обеспечивать:

- выдачу списка новостей или объектов - до 1.5 сек;
- открытие карточки - до 1 сек;
- API выдачу карты по bbox - до 1 сек;
- первичную отрисовку рабочей карты - до 2 сек.

### 20.2. Надежность

- все write-операции транзакционны;
- схема поддерживается миграциями;
- soft delete не теряет данные физически;
- ошибки API логируются;
- критические data flows покрыты тестами.

### 20.3. Эксплуатация

Система должна запускаться локально и на сервере через Docker Compose.

Обязательные сервисы MVE:

- `app`
- `postgres`

Опционально:

- `adminer` или `pgadmin`

## 22. Критерии приемки

Система считается принятой по MVE, если одновременно выполнены условия:

### По данным

- есть миграции для основных таблиц;
- seed данные загружаются одной командой;
- схема включает PostGIS и обязательные индексы;
- подготовлены минимум три BI/read-model представления.

### По объектам

- можно создать, изменить и мягко удалить объект;
- объект появляется в списке и на карте;
- у объекта открывается карточка;
- в карточке отображаются связанные новости.

### По новостям

- можно создать и изменить новость;
- новости фильтруются по периоду и источнику;
- новость отображается в списке;
- у новости открывается карточка;
- в карточке отображаются связанные объекты.

### По карте

- карта показывает объекты;
- карта показывает новости с координатами;
- фильтры синхронизируют список и карту;
- map endpoints используют bbox или эквивалентный viewport filter.

### По интеграции

- есть SQL views/read models для BI;
- schema и API documentированы;
- приложение запускается локально через Docker Compose;
- EMIS может сосуществовать с текущим BI-контуром без конфликтов namespace и контрактов.
